# Handoff Document — Self-Evolving BA Agent

## 1. What We're Building

A **self-evolving data retrieval agent** deployed on ECS that:
- **Daily morning (4:30pm China)**: Reads company KnowledgeBase → generates 10 data retrieval questions → builds SQL from skill → queries DB → emails question+SQL+results to 4 domain experts
- **Hourly heartbeat**: Checks expert replies via IMAP → classifies as correct/incorrect/unclear → evolves skill files when experts give corrective feedback → only notifies user (梁晟) if skill evolved
- **Skill is the single source of truth**: Data agent CAN ONLY use skill files for table schemas/SQL patterns. Evolver ONLY updates skill files. Closed loop.

## 2. Architecture (CRITICAL)

```
question_generator.py ──reads──→ .claude/skills/dataqueryplus/*
data_retriever.py     ──reads──→ .claude/skills/dataqueryplus/*
skill_evolver.py      ──writes─→ .claude/skills/dataqueryplus/*
```

- **data_retriever.py**: `_read_skill_context()` reads SKILL.md + data_dictionary.md + sql_templates.md fresh each call. No hardcoded table schemas anywhere.
- **question_generator.py**: Same — `_read_skill_context()` reads skill for table/field knowledge. No hardcoded schemas.
- **skill_evolver.py**: Updates the same skill files when experts give corrective feedback.

The last ECS morning run had 6/10 queries fail with `DB Error: API 错误: API执行失败!,[line 12:16~12:23 ,QIL`. 3 queries returned 0 rows (valid SQL, just no matching data).

**Likely root cause**: The skill files in `.claude/skills/` were not clean — old `data-query/` and new `dataqueryplus/` may have been mixed, or the zip was extracted over an existing directory leaving stale files. The user's colleague confirmed that `dataqueryplus.zip` is standalone and correct.

**Fix**: Delete all skill directories first, then extract `dataqueryplus.zip` cleanly. Load skill files fresh at runtime (agents already do this via `_read_skill_context()`).
